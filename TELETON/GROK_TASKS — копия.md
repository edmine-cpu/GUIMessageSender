# Задания для Grok по проекту Teleton

## Задание 1: точный аудит реальных багов

Сделай не общий обзор, а точный технический аудит по реальным багам.

Найди конкретные файлы, функции и строки кода по этим проблемам:

1. Почему циклическая рассылка не может нормально запускать несколько кампаний параллельно.
2. Где именно используется один глобальный `_cycle_thread` / `_cycle_running` / `stop_event`.
3. Почему кнопка `Stop` визуально нажимается, но задача может продолжать работать.
4. Почему кнопка `Создать` кампанию может не реагировать.
5. Почему импорт TData может добавлять аккаунт вида `tdata_879...` вместо реального номера.
6. Почему одиночный и массовый импорт TData падает “до чтения TData”.
7. Где может портиться или неполно сохраняться `socks5` proxy.
8. Где UI может подвисать из-за тяжелой работы в главном потоке.
9. Почему таблица статусов неудобная: непонятно, какой следующий чат, какой аккаунт, какая последняя ошибка.
10. Какие изменения нужны минимально, без полного переписывания проекта.

Формат ответа:

- Файл / функция / строка
- Что сейчас не так
- Как исправить
- Риск правки
- Приоритет `P0` / `P1` / `P2`

Не предлагай обход Telegram-лимитов, банов или антиспам-защит. Анализируй только стабильность приложения, UI, очереди, импорт, БД и диагностику.

## Задание 2: план минимальных правок

Составь минимальный план правок без полного переписывания проекта.

Цель: чтобы приложение стабильно работало для легальных задач, не зависало, корректно останавливало процессы, нормально импортировало TData и понятно показывало ошибки.

Раздели:

- `P0` - критично исправить сразу
- `P1` - важно, но можно после
- `P2` - улучшения интерфейса и удобства

Для каждого пункта укажи:

- какие файлы затронуть;
- почему это важно;
- какие риски;
- как проверить после правки.

## Задание 3: архитектура циклических кампаний

Проверь архитектуру циклических кампаний.

Мне нужно понять:

1. Можно ли в текущей версии безопасно запускать несколько кампаний.
2. Если нельзя, почему именно.
3. Что нужно переделать: `worker per campaign_id`, `account lock`, `task state`, `stop_event per campaign`.
4. Как сделать, чтобы один аккаунт не использовался одновременно в двух задачах.
5. Как отображать статусы в UI: текущий чат, следующий чат, аккаунт, `sent`, `err`, `last_error`, `next_attempt`.

Дай конкретные места кода и предложи минимальную схему доработки.

## Задание 4: импорт TData

Проверь импорт TData.

Найди:

1. Где выбирается папка.
2. Как программа отличает одну папку `tdata` от родительской папки с несколькими аккаунтами.
3. Почему может создаваться аккаунт `tdata_879...` вместо телефона.
4. Где должен быть preflight-check.
5. Какие ошибки нужно показывать пользователю вместо “импорт прерван”.
6. Какой лог должен сохраняться.

Формат: конкретные строки, причина, исправление, тест.

## Задание 5: сообщение разработчику

Сделай отдельный список того, что отправить разработчику.

Стиль: спокойно, профессионально, без грубости.

Нужно объяснить:

- что уже работает;
- что тестировалось;
- что не работает;
- какие баги критичные;
- какие доработки нужны;
- что желательно проверить самому разработчику перед отправкой новой версии.

Не пиши общими словами. Опирайся на найденные баги в коде и реальные симптомы.
