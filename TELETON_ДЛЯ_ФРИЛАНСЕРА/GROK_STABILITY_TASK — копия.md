# Teleton Stability Task

Работай только над стабильностью и понятностью состояния приложения. Не меняй лимиты отправки, задержки, правила Telegram, обходы ограничений или механику увеличения объема отправки.

Контекст:
- GUI: `gui.py`
- База: `data/teleton.db`
- Основная проблема пользователя: приложение показывает, что циклическая кампания запущена, но фактически может бесконечно крутить ошибки `empty_text`, кнопки Stop/Start выглядят неочевидно, несколько кампаний сложно контролировать.
- В проекте уже есть частичная архитектура `_cycle_runners: dict[str, dict]`, но остались старые общие поля `_cycle_running`, `_cycle_stop_event`, `_cycle_thread`, `_cycle_running_campaign_name`.

P0 задачи:

1. Проверить и довести до консистентного состояния запуск/остановку циклических кампаний.
   - Одна кампания = один runner по `campaign_id`.
   - Stop должен останавливать выбранную кампанию и не зависеть от того, сколько времени осталось в sleep.
   - Кнопки не должны выглядеть замороженными, если работает другая кампания.
   - Не автостартовать выключенные кампании.

2. Исправить случай `message_source='saved'`, когда у аккаунта нет текстов в Saved Messages.
   - Не гонять цели бесконечно с `empty_text`.
   - Логировать понятную причину.
   - Если у всех аккаунтов кампании пустой источник текста, остановить только эту кампанию и показать причину.

3. Улучшить статус циклической рассылки в UI без большого редизайна.
   - Отображать активные кампании списком: имя, аккаунт, позиция, sent/err, last_error, next target.
   - Нижний зеленый статус не должен залезать под кнопки.
   - Таблица целей должна быть отсортирована так, чтобы текущая/следующая цель была сверху.

4. Проверить импорт TData/сессий.
   - Нельзя добавлять псевдо-аккаунты вида `tdata_*`.
   - При ошибке импорта показывать конкретную причину, а не только “подробности в логах”.

5. Проверить DB/schema.
   - Не должно быть выбранных аккаунтов кампании, которых нет в `accounts`.
   - Если такие есть, добавить безопасную очистку/валидацию или понятное предупреждение.

Ограничения:
- Не переписывать весь проект.
- Не менять бизнес-логику рассылки, задержки и лимиты.
- Все изменения должны быть минимальными и проверяемыми.
- После правок запустить:
  `py -3.12 -m pytest tests/ -q`
- В ответе указать:
  1. измененные файлы;
  2. что конкретно исправлено;
  3. результат тестов;
  4. что осталось риском.
